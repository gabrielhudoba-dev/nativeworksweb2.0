SERVICE SCHEDULE — DESIGN SYSTEM

Annexed to SOW Ref: {{sow_ref}}

─────────────────────────────────────────────────

D1. SCOPE

This Schedule covers the creation of a comprehensive design system — a shared library of design tokens, components, patterns, and documentation — as described in the Proposal.

D2. DELIVERABLES

     (a) Design tokens: colour scales, typography ramp, spacing scale, radius, shadow, motion tokens
     (b) Base component library in Figma: atoms (Button, Input, Checkbox, Tag, Badge, Avatar, Icon, etc.)
     (c) Composite components: Navigation, Modal, Dropdown, Card, Form, Table, Toast/notification
     (d) Pattern library: page layouts, data entry flows, empty states, error states
     (e) Component documentation: usage guidelines, dos/don'ts, accessibility notes, variant descriptions
     (f) Handoff: Figma library file set up as a published library in Client's Figma organisation
     (g) (Optional — if specified) Front-end component implementation in agreed framework

D3. NOT INCLUDED (REQUIRES CHANGE ORDER)

     — Illustration system or iconography creation (existing icon libraries may be integrated)
     — Animation / motion component design beyond transition specifications
     — Implementation in multiple front-end frameworks
     — Ongoing governance or component update services (covered under Retainer Schedule)
     — Documentation website or Storybook implementation (unless specified in Proposal)

D4. COMPONENT SCOPE

The specific components included in this design system are:

     Core UI:          As listed in the Proposal
     Page templates:   As agreed during discovery
     Custom:           Any components beyond the listed scope require a Change Order

D5. REVISION ROUNDS

     Token system / foundations:    One (1) round of revisions
     Component designs:             Two (2) rounds of revisions per component tier
     Documentation:                 One (1) round of revisions

D6. ACCESSIBILITY

All components shall be designed to meet WCAG 2.1 Level AA accessibility requirements at the design level. Native Works makes no warranty regarding the accessibility of front-end implementations performed by third parties after handoff.

D7. GOVERNANCE

Native Works recommends that Client designate an internal design system owner responsible for maintaining the system after handoff. Native Works can provide a governance framework document as part of the documentation deliverable. Long-term maintenance is available under a Retainer Schedule.

D8. OWNERSHIP

All design tokens, components, and documentation created under this Schedule are Work Product and transfer to Client upon full payment, subject to the Background IP provisions of the MSA. Figma file ownership is transferred to Client's Figma organisation upon final payment.
